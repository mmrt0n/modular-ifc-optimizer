# 모듈러 자재 배치 시스템 — 코드 통합 핸드오프 프롬프트

> 이 문서는 다른 Claude Code 인스턴스에게 작업 맥락을 정확히 전달하기 위한 프롬프트입니다.
> 아래 내용을 그대로 붙여넣고 작업을 시작하세요.

---

## 0. 너의 역할

너는 건축 BIM(IFC) 기반 **경량벽체 석고보드/합판 배치 최적화 시스템**의 코드를 통합하는 작업을 한다.
서로 다른 팀원이 만든 두 코드베이스를, 아래에서 **이미 확정된 결정**에 따라 하나로 합쳐야 한다.
**확정된 결정은 다시 묻지 말고 그대로 적용**하고, "미정(TBD)"으로 표시된 것만 사용자에게 확인해라.

---

## 1. 프로젝트 개요

- **목적**: IFC 파일에서 경량벽체 정보(길이·높이·층·공간·개구부)를 추출 → M3시스템즈 실제 시공방식(각재 450mm · 1P · 세로)으로 석고보드(900×1800)/합판(1220×2440) 배치를 자동 계산 → 직선절단·노치절단·발주수량·로스율 산정 + 자투리 재사용.
- **맥락**: 현장 시공이 아니라 **모듈러(공장 제작)** 방식. 즉 벽이 건물 층 단위가 아니라 **공장에서 모듈(박스) 단위로 제작**된다.
- **자재**: 석고보드 900×1800mm / 합판 1220×2440mm. 각재 간격 450mm. 1P 단겹. 세로(종방향) 시공. 바닥 밀착.

## 2. 두 코드베이스

### (A) 동영 EXE = **통합의 베이스(뼈대)** — Python
- 저장소: GitHub `mmrt0n/modular-ifc-optimizer`
- 파일:
  - `ifc_verifier.py` — IFC 파싱·검증(이중검증/교차검증), 벽/개구부/치수 추출
  - `gypsum_optimizer_v3.py` — 배치 최적화 엔진 + 재사용 풀(ReusePool) + HTML 생성
  - `ifc_pipeline.py` — Tkinter 마법사 + 전체 파이프라인 오케스트레이터(EXE 진입점)
- 강점: **IFC 파싱·검증 인프라**, **재사용(ReusePool) 구현됨**, 발주/검증 HTML 산출.
- 한계: 벽을 평면 L×H로만 처리(돌출/꺾임 미처리), 개구부 보드를 `cut_op` 한 덩어리로만 분류(노치 구분 없음), 재사용이 층 단위로 제한됨, 평가 JSON 없음.

### (B) 승훈 HTML 시뮬레이터 = **좋은 조각만 이식해올 대상** — 단일 HTML+JS
- 파일: `승훈 시뮬레이터 코드(vs code로 열기).html` (단일 벽 시각화/애니메이션 도구)
- 강점: **노치 분류**(개구부가 보드 내부 관통 시 ㄱ/ㄴ자 2회 절단), **개구부 코너 이격 경고(200mm)**, **평가지표 JSON V3**(schema=M3_BOARD_EVALUATION_RESULT, version 3.0), 세로/가로 자동방향, 시공 애니메이션.
- 한계: IFC 파싱 없음(벽 데이터 직접 입력), **재사용 미구현(reuseSheets 항상 0)**.

### (C) 이승훈 워드 문서 = 사양서
- 파일: `석고보드,합판 시공 및 재사용로직 정리.docx`
- 내용: (B)의 시공 로직(3장)·평가 로직(5장)을 글로 정리 + **재사용 판정기준(4장, 아직 미구현 → (A)에 넣어야 할 사양)**.
- 재사용 사양 핵심: 최소규격 **폭 300mm & 높이 450mm 이상**, 규격 일치 불필요, **위치 제한 없음**(벽면 어디든·개구부 내부 포함), 자투리 발생→규격검사→창고저장→다음 벽/부족구간 우선적용→재고차감.

## 3. 통합 방향 (확정)

**베이스 = 동영 EXE(Python).** 승훈 HTML에서 아래 4개를 EXE로 이식한다.
권장 최종 구조: **EXE = 계산 엔진(평가 JSON 출력) / 승훈 HTML = 그 JSON을 읽어 시각화만 담당** (역할 분리).

### 승훈 → EXE로 이식할 것
1. **노치 분류**: 승훈 `classify()`의 throughX/throughY 판별(개구부가 보드 내부 좌우/상하 관통 → notch)을 EXE에 포팅.
2. **개구부 코너 이격 경고**: 이음매가 개구부 모서리 200mm 이내면 경고 카운트(`CORNER_MIN=200`).
3. **평가지표 JSON V3** 출력: schema=M3_BOARD_EVALUATION_RESULT, version 3.0 (벽단위/프로젝트단위). 두 시스템 결과 자동 대조용.
4. (자동 방향은 **이식하지 않음** — 세로 고정으로 간다.)

## 4. 충돌 해소 — **확정된 결정 (다시 묻지 말 것)**

| 충돌 항목 | 확정 결정 | 구현 시 주의 |
|---|---|---|
| **개구부 x좌표** | **EXE 기준(왼쪽 끝) 사용** | 승훈 HTML에 데이터 줄 때만 중심좌표(`x + ow/2`)로 변환. EXE의 `export_simulator_walls`에 이미 변환 있음 → **이중변환 금지** |
| **바닥 이격** | **0 (바닥 밀착)** | 승훈 쪽 `FLOOR_GAP=5`를 **0으로** 바꿔야 두 시스템 행(row) 수가 일치. M3 "바닥 밀착"과 일치 |
| **시공 방향** | **세로(종방향) 고정** | 승훈의 세로/가로 자동선택은 제거 |
| **절단 분류 체계** | **2축으로 분리** | ① 절단형상: `full / edge_cut / notch / skip` (승훈 것 이식) ② 출처: `new / reuse` (EXE 것 유지). 한 enum에 섞지 말 것 — "재사용한 노치"도 표현 가능해야 함. 평가 JSON의 `notchCutSheets`/`reuseSheets` 둘 다 정확히 채워지도록 |
| **재사용 범위(scope)** | **공장(생산 배치) 단위** | 현재 EXE는 `space_id`(방)→`floor_id`(층)로 제한. 이를 **공장 배치 단위**로 교체. 층 경계 제거 |
| **재사용 위치(position)** | **제한 없음** | 워드 4장 사양: 벽면 어디든·개구부 내부 포함 적용 가능. ※ "범위(scope)"와 "위치(position)"는 별개 개념 — 따로 구현 |

## 5. 미정 (TBD) — 사용자 확인 필요

1. **"공장(배치) 단위"를 데이터에서 어떻게 식별하나?**
   - 현재 EXE엔 `space_id`, `floor_id`만 있고 **모듈/공장 식별 필드가 없다.**
   - **공장 1배치당 모듈을 몇 개 만드는지** 기준이 있어야 재사용 풀 경계를 정할 수 있다. (사용자가 이 기준을 알아본다고 함.)
   - IFC에 모듈/유닛 속성이 있는지 확인 필요. 없으면 공간(space) 그룹핑 규칙을 정의해야 함.
   - ⚠️ 워드 4.5장 경고: "타 공장(외주) 활용 시 공장 간 자재 이동/물류는 본 기준에서 제외" → 풀 경계를 실제 공장 생산 단위와 맞춰야 함(너무 넓게 잡으면 못 옮기는 자투리를 재사용으로 오집계).

2. **배치 시작 방향 (RTL vs LTR)**: 누적공차를 어느 끝에서 흡수하는지. EXE는 개구부 없는 벽에 RTL(오른쪽부터), 승훈은 LTR 고정. **M3 실시공 기준을 사용자가 확인 중.**

## 6. 추가 참고 (워드 부록 A — 알려진 이슈)
- 합판 팔레트 단위가 코드 함수마다 상이(990/100/50장) → 실제 출고 단위로 통일 필요.
- 돌출벽/꺾인 벽: 현재 두 시스템 모두 벽을 평면 L×H로만 처리 → 추후 개선 항목.

## 7. 작업 시작 시 권장 순서
1. **Phase 1 (수치 정합)**: 같은 벽 1개를 두 시스템에 넣고 보드수·로스율 비교 → 충돌 6개 중 무엇이 수치를 틀어지게 하는지 데이터로 확인 → 상수/좌표계 통일. 평가 JSON V3를 EXE에도 출력해 자동 대조.
2. **Phase 2 (분류 이식)**: 노치 분류 + 코너 경고를 EXE에 포팅. `cut_op` → `edge_cut`/`notch` 세분화 + 출처 축 분리.
3. **Phase 3 (재사용 정합)**: ReusePool을 공장 배치 단위 + 위치 무제한으로 변경(단, 5번 TBD 확정 후).
4. **Phase 4 (통합 산출)**: EXE가 검증 + 시공(노치) + 재사용 + 평가 JSON을 한 번에 출력. 승훈 HTML은 그 JSON을 읽어 시각화만.
